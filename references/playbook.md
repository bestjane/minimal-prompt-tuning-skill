Prompt / Tool 调优方法论（精简版）

适用范围：LLM 意图识别、工具调用、参数抽取、多步骤执行、历史承接、对话收口、评测脚本和测试集维护。

核心原则：先看真实链路，再做最小改动；先判断根因，再决定改哪里；prompt 管流程和优先级，tool/schema 管边界和字段，测试脚本管统计口径，测试集管期望。


一、标准调优闭环

每轮按这个顺序做：
1. 跑全量或抽样，拿到 badcase。
2. 抽出可复现小集。
3. 看 raw log，不只看状态标签。
4. 判断根因：流程、边界、参数、历史、多意图、工具返回、统计、网络、标注。
5. 选修改位置：prompt、tool/schema、测试脚本或测试集。
6. 做最小修改，优先改已有同类条目。
7. 只跑对应 badcase 小集。
8. 小集收敛后，回跑本轮受影响集合。
9. 有网络或 HTTP 错误时先重跑，不当成 prompt 问题。
10. 若改动影响面大，再跑抽样或全量。

每轮必须能说明：
- 失败的真实链路是什么。
- 根因是什么。
- 为什么改这里。
- 影响范围是什么。
- 哪些 case 已验证。


二、看 log 的最小清单

优化前至少确认：
- 用户输入和有效历史是什么。
- 模型第一段输出是什么。
- 有没有真实 tool_call / function_call / action。
- 调用了哪个工具，顺序是否正确。
- 参数是否完整，字段是否符合 schema。
- 工具返回了什么。
- 工具返回后模型是否继续误触发。
- 工具返回中的疑问、反问、澄清、候选在最终回复中是否保留原句式。
- response_text 是否只是函数名、JSON 或伪 action。
- 失败来自模型、工具、网络、统计脚本，还是测试集标注。

同一个状态可能有不同根因：
- `no_tool_call` 可能是漏调用，也可能是伪工具调用。
- `argument_mismatch` 可能是字段说明不足，也可能是测试期望不合理。
- `unexpected_tool_call` 可能是流程优先级错，也可能是工具边界太宽。
- `success` 只说明通过当前口径，不代表过程一定正确。


三、根因决定修改位置

按下面规则选位置：

- 流程错，改 prompt。
  - 应调用却停在自然语言。
  - 应澄清却执行。
  - 多步骤顺序错。
  - 工具返回后循环调用。
  - 建议、澄清、不支持、执行的优先级不清。

- 工具边界错，改 tool description。
  - 该命中某工具却没命中。
  - 不该命中某工具却误命中。
  - 相邻工具职责交叉。
  - 某类表达属于哪个工具不清楚。

- 参数错，改 schema 字段 description。
  - 字段缺失。
  - 枚举选错。
  - 数值、单位、格式错。
  - 默认补值或字段归一化不稳定。
  - 模型理解了意图，但填参时忘了规则。

- 伪工具调用，改 prompt，必要时同步改高风险 tool。
  - 正文输出函数名。
  - 正文输出参数 JSON。
  - 正文输出 action 文本。
  - 看起来像调用了，但 actual_tool_calls 为空。

- 历史承接错，优先改 prompt。
  - 有效上下文、关闭条件、可继承内容不清楚。
  - 若历史补全后字段仍缺，字段规则同步写 schema。

- 统计错，改评测脚本。
  - 流式事件解析错。
  - 工具顺序或伪调用识别错。
  - 网络错误和模型错误混在一起。

- 标注错，改测试集或统计口径。
  - 不要为了迎合错误标注污染 prompt/tool。

快速判断：
- 回复理解正确但参数错，多半改 schema。
- 调了相邻工具，多半改 tool 边界。
- 应澄清却调用，多半 prompt 和 tool 边界都要收紧。
- 正文像函数调用但没有 tool_call，是通道问题。


四、规则写法

每条规则都写成：触发条件 + 正确动作 + 禁止项。

有效写法：
“当 A 且 B 时，必须做 C；不得做 D；本轮最多 E。”

无效写法：
“合理处理、尽量避免、注意不要、保持自然、必要时处理。”

优先用硬约束：
- 必须。
- 不得。
- 禁止。
- 只。
- 本轮最多。
- 同一子任务最多。
- 只能来自真实用户输入或有效上下文。

如果两条规则可能冲突，必须写先后关系：
- 先判断什么。
- 再执行什么。
- 命中冲突时谁优先。

缺槽规则必须同时写：
- 必须澄清。
- 禁止把模糊值写进字段。
- 禁止删除模糊片段后按立即执行。
- 禁止默认补不存在的对象、动作、数量、时间、范围或目标值。


五、Prompt 应该写什么

Prompt 负责“模型怎么工作”。适合写：
- 角色职责和能力范围。
- 每轮处理流程和优先级。
- 什么时候调用工具、什么时候澄清、什么时候建议、什么时候直接答。
- 多意图如何拆分、排序、收口。
- 工具返回后什么时候停止、什么时候继续。
- 历史跟进如何补全。
- 元问题、规则讨论、能力讨论不执行其中控制词。
- 承接语、最终回复、失败回复的口径。
- 命中工具时必须真实 tool_call，禁止正文输出函数名或 JSON。

Prompt 不适合：
- 大量复述所有工具字段。
- 为每个工具堆参数格式。
- 用长篇解释代替硬规则。
- 把单个 badcase 补丁放在末尾。


六、Tool / Schema 应该写什么

Tool/schema 负责“这个工具处理什么、字段怎么填”。适合写：
- 本工具命中范围。
- 本工具明确不命中的输入。
- 与相邻工具的边界。
- 动作枚举映射。
- 参数字段含义、格式、必填条件、禁填条件。
- 单位和值如何拆分。
- 目标值、端点值、过程型动作如何对应枚举。
- 本工具特有的高风险短约束。

Tool/schema 不适合：
- 复制完整全局流程。
- 长篇解释业务策略。
- 在每个工具里重复同一套 prompt。
- 只写“遵守全局规则”而不保留字段锚点。

字段级高风险锚点可以很短，但不能完全删除：
- 何时必填。
- 何时禁填。
- 何时保留原文。
- 何时不能归一化。
- 何时不能默认补值。

Tool 越简洁，动作和枚举边界越要精准：
- 目标型动作对应 `set`。
- 过程型动作对应 `adjustUp/adjustDown`。
- 开关型动作对应 `open/close`。
- 查询型动作对应 `query`。
- 端点值对应 `min/max/mid`。


七、全局规则和局部锚点要分层

不要二选一：
- Prompt 写完整规则、流程和优先级。
- Tool/schema 写本工具、本字段最容易出错的短约束。

不要在 tool 里复制整套 prompt。
也不要把高风险字段只写成“按全局规则”。

判断一个局部约束能不能删：
- 删除后模型是否还知道字段怎么填。
- 删除后模型是否还知道什么时候不能调用该工具。
- 删除后模型是否还知道相邻工具边界。
- 删除后是否会把模糊槽位写进参数或默认补值。

不确定时，不要完全删除，压缩成一句字段锚点。


八、缺槽、模糊和工具吸引力

如果输入缺对象、动作、数量、时间、范围、查询目标等关键槽位：
- 能从有效上下文唯一补齐，才允许补全后调用。
- 不能唯一补齐，必须澄清。
- 不得把模糊值写进参数。
- 不得忽略模糊片段后执行剩余意图。
- 不得默认补产品未定义的对象、动作或值。

如果某个工具容易误吸模型：
- Prompt 写“缺槽必须澄清”。
- 对应 tool 写“缺槽不命中本工具”。

不要为了通过测试扩大不自然的工具边界。
缺对象、缺动作、无上下文的孤立片段，不应强行路由到业务工具。


九、多意图、工具返回和循环

多意图有两个独立约束：
- 顺序：按用户原句顺序调用，不重排。
- 中间输出：多个工具串行时，中间不得输出文本。

“最多一次”要按子任务定义：
- 同一子任务最多一次。
- 不同子任务可按顺序分别调用。
- 不得因工具返回或模型自造文本新增子任务。

工具返回后是否继续调用，只看来源：
- 后续调用来自当前真实用户输入中尚未执行的明确子任务，可以继续。
- 后续调用来自工具返回文本、模型自造问句、模型替用户确认，禁止继续。
- 工具返回里的问题、建议、候选是给用户看的，不是给模型自己回答的。
- 工具返回里的疑问或反问必须保留疑问或反问句式，不能改成陈述句、执行承诺或执行结果。
- 不重复调用和正确转述要分开验证：tool_call 次数正确，不代表最终回复没有改变工具返回的权限、语气或状态。

修循环不要只写“不要循环”。必须说明循环来源和停止条件。


十、历史跟进

历史规则必须定义四件事：
- 哪类历史是有效上下文：未关闭的建议、澄清、候选、询问、未完成任务、最近明确查询对象。
- 哪类用户回复能接上：确认、拒绝、指代、补全、序号、只补时间/数量/方向/目标值等。
- 什么时候上下文关闭：任务完成、用户拒绝、用户切到无关话题且无指代、新请求覆盖旧请求。
- 跟进调用时如何补全：必须合并上下文和本轮信息，不能只传“好的/可以/就这样”等片段。

同一个模糊短语要区分位置：
- 同句附带的模糊请求，通常不能默认补对象。
- 独立一轮的跟进请求，可以继承有效历史。
- 没有效历史时，必须澄清。
- 有明确对象时，直接执行或查询。

不要简单写“上一轮是什么就继承什么”。


十一、话术和元问题

承接语不能单独成为最终回复：
- 命中工具时，承接后必须立即真实 tool_call。
- 承接语不得包含执行结果、虚假进度、支持判断或结论。
- 不得只说“我帮你看一下”就结束。

不要按词全局禁用，要按用途禁用：
- 准备处理的话术可以保留。
- 执行结果、执行承诺、虚假完成的话术要禁止。
- 黑名单只打高风险用途，不要把正常表达材料都删光。

承接语调优不要依赖正向样例池：
- 固定样例会成为高频模板，尤其是 prompt、persona 和 history 里的 assistant 话术。
- 优先写承接语目的、信息来源和禁止改变语义。
- 允许表达也要避开高频诱导词；某个词是重复源时，不要放进允许列表。
- 规则不能只写“不要怎么说”，还要保留可说空间：贴用户原话取锚点，只表达准备处理，工具后再转述结果。

话术重复要统计句式家族：
- 不只看完全相同句子。
- 同时看开头模式、人称组合、高频动词、固定骨架和尾句。
- 精确重复不高，不代表用户听感不重复。

元问题、规则讨论、能力讨论不执行其中控制词：
- 用户问“如果我说 X 你会怎么处理”，直接答规则，不执行 X。
- 用户问“支持什么”，直接答能力边界。
- 除非用户明确改为真实任务请求。


十二、样例和边界词

不要靠堆例子修问题。
优先抽象成语义簇：
- 同一业务场景。
- 同一槽位缺失。
- 同一工具边界。
- 同一参数格式。
- 同一历史承接模式。

具体样例要尽量变量化：
- 写“某公司当前 CEO”，不要写死某家公司。
- 写“城市 + 明天 + 限行尾号”，不要写死城市。
- 写“只补时间 + 时长”，不要写死具体时间。

history 里的 assistant 内容也是样例：
- 如果历史中大量出现同一句承接语，模型会把它当风格标准。
- 构造测试语料时，避免让 history 反复强化同一话术骨架。
- 分析话术重复时，同时检查 prompt、persona、tool 和测试语料 history。

短负例可以用，但必须满足：
- 属于稳定语义簇。
- 写在对应工具或字段里。
- 只服务一个高风险边界。
- 足够短，不替代规则本身。

边界词有歧义时，要拆成业务含义：
- 查询含义和控制含义分开。
- 助手自身问题和外部事实问题分开。
- 音量、播放、设备控制等相邻能力分开。


十三、显式信息、字段分支和共享规则

先区分信息来源：
- 用户明确说出的值，必须保留。
- 有效历史能唯一补齐的值，可以继承。
- 模型自己推出来的默认值，必须受限。
- 禁止默认补值时，不得误伤用户明确给出的值。

不要按单个关键词判断：
- 先看完整句式和槽位关系。
- `动作 + 到/至 + 目标值` 通常是目标型设置。
- 过程型动作只有在没有目标值时，才按增减处理。
- 同一个词在查询、控制、时间、范围里可能含义不同，必须按结构分开。

同一字段必须按模式分支：
- 先判断字段处在哪种模式，再决定保留、删除、禁填或继承。
- 写了周期字段时，时间字段通常删除周期词，只保留时段和钟点。
- 没写周期字段时，一次性日期词通常必须保留。
- 裸结束时点是否继承日期，必须由开始时点模式决定。

字段规则要成对写：
- 写“什么时候填”，也写“什么时候不填”。
- 写“什么时候保留”，也写“什么时候删除”。
- 写“显式值怎么填”，也写“默认值何时禁止”。
- 写“共享字段通用规则”，也写“本工具特有例外”。

共享字段改动要扩大验证范围：
- 时间、数值、单位、周期、查询对象、历史承接属于高风险共享字段。
- 修改共享字段后，必须回跑同字段、同流程、同边界的旧 badcase。
- 只跑当前失败样本不够；要补跑受影响集合，防止反向坏点。


十四、结构和精简

不要补丁式修复。
补丁式表现：
- 末尾不断新增“补充规则/特殊情况/再次强调”。
- 每个 badcase 加一句例外。
- 同类规则散落多处。
- 新规则只解决当前样本，破坏结构。

有效做法：
- 先按流程、边界、参数、历史、多意图、工具返回、统计口径归类。
- 优先修改已有同类条目。
- 同类规则太散时，先合并重构，再补细节。
- 新增规则必须能覆盖一组同类问题。

精简前先做“信息量账本”：
- 把原规则拆成独立约束点。
- 每个约束点在新版本中找到位置。
- 可以合并表达，不能删除语义。
- 可以改成命令式，不能改窄或改宽条件。
- 改完回跑原 badcase 小集，确认没有反向坏点。

好的精简不是信息更少，而是同样信息更少歧义。


十五、验证和统计

不要盲信报告，必要时看 raw response / raw events。

评估时分层看：
- 路由是否正确。
- 是否真实 tool_call。
- 是否伪调用。
- 工具顺序是否正确。
- 参数是否完整准确。
- 是否循环。
- 工具返回后是否收口。
- 最终回复是否基于真实工具结果。
- 最终回复是否保留工具返回的疑问、反问、澄清或候选形态。

测试模式按问题选择：
- 看承接语，检查首个 tool 前文本。
- 看重复调用，检查完整 raw events 和 tool_calls 数量。
- 看最终转述，检查完整 response_text。
- 看话术随机性或模板复发，同一句重复跑多次。

网络和远程接口错误：
- http_error、network_error、502 先重跑。
- quota/token-limit、partial stream、中途断流可能被统计成参数错、答案错或工具数量错，必须看 raw 定性。
- 重跑成功，不改 prompt。
- 重跑后仍是模型错误，再定位根因。
- 小集收敛时，网络错误不计入 prompt 修复失败。

测试脚本也可能错：
- 解析错，改脚本。
- 统计口径错，改脚本。
- 标注不符合产品规则，改测试集。
- 不要用 prompt 修脚本或标注问题。


十六、反向坏点和停止条件

每次加规则前先问：
- 会不会让无关输入误进工具。
- 会不会让明确任务被错误澄清。
- 会不会阻断合法多步骤。
- 会不会引起循环调用。
- 会不会让模型只输出承接语或伪调用。
- 会不会默认补不存在的值。
- 会不会和已有规则冲突。

如果修复引入反向坏点，先判断：
- 规则是否过宽。
- 规则是否放错位置。
- 规则是否缺少优先级。
- 产品策略是否没定清楚。

应该停止优化的情况：
- 剩余问题是标注争议。
- 修复会扩大不合理工具边界。
- 修复需要大量样例补丁。
- 小集收益小但风险大。
- 规则开始互相冲突。
- 问题来自随机波动且无法稳定复现。

应该继续优化的情况：
- 明确流程错。
- 明确边界错。
- 明确参数错。
- 明确脚本错。
- 明确存在可抽象语义簇。

最终目标不是让模型更自由，而是让模型少自由发挥：
- 命中执行条件就真实调用工具。
- 缺关键信息就澄清。
- 参数只来自输入或有效上下文。
- 工具返回后按规则收口。
- 多步骤按用户原句顺序。
