目标：在尽量少增加 token 的前提下，提升意图判断、工具路由、参数提取和回复稳定性，避免为少量 badcase 堆叠 prompt。

一、总原则
1. 优先做“压缩后仍有效”的优化，不做靠堆文案换效果的优化。
2. 先判断问题属于“路由错、参数错、回复错、退出误判、建议/执行边界错”哪一类，再定点修。
3. 能改 `tools_call.json` 解决的，不改 `system_prompt.txt`；能改单个 tool 描述解决的，不加全局规则。
4. 新增约束必须满足 3 个条件：高频、稳定、可泛化。只覆盖个别语料的规则不加。
5. 所有新增话术都要短句化，避免同一信息在 prompt、tool 描述、参数描述里重复出现。

二、推荐调优顺序
1. 先看 badcase 是否能归入已有规则。若只是模型没遵守，不要立刻加新规则，先检查现有表述是否重复、分散或不够靠前。
2. 若问题是某个工具经常误路由，优先补该 tool 的“用途边界”，不要在总 prompt 里加长篇说明。
3. 若问题是某个参数经常漏填、错填，优先补该参数描述，不要把参数细节再写回 tool 顶层描述。
4. 若问题涉及多个工具共享规则，再考虑放入 `system_prompt.txt` 的对应模块。
5. 若问题只在极少数语料出现，优先接受误差，或等更多样本后再决定是否加规则。

三、问题归因与最小改法
1. 路由错
优先改 tool 顶层 `description`，只写“这个工具管什么，不管什么”。
不要重复写参数枚举、JSON 结构、固定值映射，避免 description 变长后反而稀释边界。

2. 参数错
优先改参数 `description`，只补该参数最关键的取值条件、保留条件、必填条件。
同一信息只保留一处。例如模式映射放在 `attrValue`，不要在 tool 顶层再重复一遍。

3. 建议和执行混淆
这类问题优先放在 `system_prompt.txt` 的主体流程或执行原则里，因为它影响多个浴霸工具。
规则要写成判定句，不要写成长解释。例如“仅描述场景且未明确功能时，先建议并确认，不得直接调用工具”。

4. 退出误判
只在前置拦截里修，不要把退出规则散落到别的章节。
退出规则只解决“什么算退出、什么不算退出”，不要混入媒体控制、工具调用细节。

5. 回复风格或播报异常
优先放在对话准则中统一约束，不要给每个 tool 单独补回复文案要求。

四、prompt 优化准则
1. 全局规则只保留跨工具、跨场景共用的内容，如退出、建议/执行边界、时间解析、查询与调节总规则。
2. 工具专属规则尽量下沉到 `tools_call.json`，避免全局 prompt 变成工具手册。
3. 一条规则只表达一个核心判断，避免“条件+例外+解释+举例”塞进同一句。
4. 能靠章节位置解决的，不靠加字解决。高优先级、易误判规则应前置。
5. 少举例，除非该映射非常不直观且没有更短写法。

五、tools 描述优化准则
1. tool 顶层 `description` 只保留两类信息：用途、边界。
2. 参数 `description` 只保留三类信息：取值含义、何时必填、何时保留。
3. 单值 `enum` 能表达清楚时，不再写“固定值：xxx”。
4. 同一个含义不要同时写在 tool 描述和参数描述里。
5. 多个工具共用的规则，不要在每个 tool 里各写一遍，能放系统 prompt 就放系统 prompt。

六、哪些情况不建议加 prompt
1. 只是某次模型随机失误，且没有形成稳定错误模式。
2. 该规则只能覆盖 1 到 2 条测试语料，泛化价值低。
3. 该信息已在其他位置表达过，只是重复一份。
4. 该问题本质是测试集答案过严，不是模型能力真的有缺陷。

七、测试驱动的优化流程
1. 每轮只改一小类问题，如退出、定时、建议执行边界、某个 tool 路由，不要一轮混改太多。
2. 改完先跑批测，看整体指标和易错集变化，再看单条 badcase。
3. 若某条规则提升了少量 badcase，却让更多正常样本变差，应回退或改写。
4. 记录每轮改动点、影响范围、收益和副作用，避免重复加同类规则。

八、具体执行建议
1. 优先处理高频错误，而不是最刺眼的个例。
2. 优先处理“一个改动能覆盖一类错误”的问题，例如退出误判、场景建议误执行、时间字段漏填。
3. 对浴霸类能力，优先保证“是否该调工具”和“工具是否选对”；其次再优化参数细节和回复风格。
4. 若必须新增规则，默认先写最短版；只有最短版压不住错误时，才补第二层限定。

九、文件落点
1. `system_prompt.txt`
放跨工具的总流程、全局判定、建议/执行边界、时间总规则、对话准则。
2. `tools_call.json`
放单个工具的用途边界、参数保留条件、枚举映射、必填条件。
3. 不要在两个文件里重复维护同一条细节规则。

十、最后要求
1. 每次优化默认先删重复，再补缺失。
2. 新增内容必须比它解决的问题更“值”；如果加了很多字只修一个小问题，通常不是好优化。
3. 最终目标不是 prompt 更长，而是结构更稳、边界更清、平均效果更好。

十一、每轮优化执行模板
1. 先跑测试，记录 4 项：总体通过率、各测试集通过率、高发错误类型、代表性 badcase。
2. 先分类，不急着改。每条 badcase 先标成以下之一：路由错、参数错、退出误判、建议误执行、该执行未执行、回复不合规、测试标准过严。
3. 看覆盖面。若 1 条改动能覆盖一类错误，优先处理；若只能修单条样本，先暂缓。
4. 定修改位置：
路由边界错，改 `tools_call.json` 的 tool 顶层 `description`。
参数漏填或错填，改对应参数 `description`。
多工具共用判定错，改 `system_prompt.txt`。
回复风格或播报问题，改对话准则。
5. 控制改动量。每轮尽量只改一类问题，避免同时改很多位置导致无法归因。
6. 改完立刻复测。先看整体是否提升，再看目标 badcase 是否改善。
7. 若局部提升但整体下降，优先回退；若整体提升但个别样本波动，可继续观察下一轮。

十三、改动决策速查
1. 用户意图经常走错 tool：优先改 tool 顶层边界。
2. tool 选对了，但字段经常缺失或补错：优先改参数描述。
3. “建议”和“直接执行”总混：优先改主体流程或执行原则。
4. “暂停/继续/切歌”被误判成退出：只改前置拦截。
5. 只是回复啰嗦、生硬、暴露内部实现：改对话准则，不改工具描述。
6. 只是某个 badcase 不过，但没有稳定模式：先不加规则。

十四、调优工作流 SOP
1. 固定基线
先保存当前版本的 `system_prompt.txt`、`tools_call.json` 和测试报告，后续所有改动都基于同一基线对比。

2. 跑一轮测试
记录总体通过率、各子集通过率、Top badcase、主要错误类型占比。

3. 先聚类，再选目标
把 badcase 按错误类型聚类，不按单条语料逐个修。
每轮只选 1 类最值得修的问题，优先选高频、可泛化、能一改多得的问题。

4. 判断落点
若是工具选错，改 tool 顶层边界。
若是参数缺失、模式漏保留、时间字段错，改参数描述。
若是建议/执行、退出、全局流程判断错，改 `system_prompt.txt`。

5. 先删重复，再补缺失
修改前先检查相关信息是否已在别处写过。
若已写过，优先挪位置、合并或前置，不先加新句子。

6. 默认最小改动
每次只改必要的几句。
能改一个 tool，不改一组 tool。
能改一个参数，不改整个章节。

7. 改后立即复测
先看目标问题是否改善，再看总体是否变差。
若目标变好但副作用明显，说明规则过宽，需要缩窄边界。

8. 做保留判断
同时满足“目标问题改善”和“整体无明显副作用”才保留。
若只修了个别样本，但拉低整体，直接回退。

9. 进入下一轮
保留当前最优版本，继续处理下一类高频问题，不在同一轮里连续堆规则。

十五、单轮执行清单
1. 读取最新测试报告。
2. 列出前 5 个最有代表性的 badcase。
3. 给每个 badcase 标错因。
4. 统计哪一类最多。
5. 判断这类问题该改 prompt 还是 tool。
6. 先检查现有规则是否已覆盖。
7. 若未覆盖，补最短新规则。
8. 若已覆盖但执行不稳，改位置、拆句或去重。
9. 复测并记录收益、副作用。
10. 决定保留、回退或继续微调。

十六、优先级建议
1. 第一优先级：退出误判、建议误执行、tool 误路由。
2. 第二优先级：时间解析错、模式或限定词漏保留、该执行未执行。
3. 第三优先级：回复冗长、播报不自然、风格不稳。
4. 第四优先级：少量边缘 badcase。

十七、控制 token 增长的硬规则
1. 新加一句前，先删一句重复或低价值表述。
2. 同一信息最多保留一处。
3. 枚举映射尽量留在参数描述，不在 tool 顶层重复。
4. 全局规则不写工具细节，工具描述不写全局规则。
5. 只有当现有最短规则压不住问题时，才允许补第二句。
