# Minimal Prompt Tuning

用这个 skill 在不为少量 badcase 堆 prompt 的前提下，提升整体稳定性。

改动要尽量小、尽量局部、尽量可归因。优先修高频、稳定、可泛化的一类错误。

## 按这个流程做

1. 先看最新测试报告，列出代表性 badcase。
2. 给每条 badcase 归类：路由错、参数错、退出误判、建议误执行、该执行未执行、回复不合规、测试标准过严。
3. 按错误类型聚类，优先处理“一个改动能覆盖一类错误”的高频问题。
4. 先检查现有规则是否已经覆盖。若已覆盖，优先前置、去重、拆句或收紧，不急着新增。
5. 只改最小且正确的落点：
   - 工具选择边界：改 `tools_call.json` 里的 tool 顶层 `description`
   - 参数填充、保留、必填条件：改对应参数 `description`
   - 跨工具共用判断，如退出、建议与执行边界、全局时间规则、回复风格：改 `system_prompt.txt`
6. 默认先写最短版规则，一句只表达一个核心判断。
7. 改完立刻复测。只有目标问题改善且整体没有明显回退时才保留。
8. 记录改动点、预期收益、实际收益和副作用。

## 放置规则

- 某个工具经常误路由，先改 `tools_call.json`
- tool 选对了但字段常漏填、错填、乱保留，先改参数描述
- 多个工具共用的判断规则，才改 `system_prompt.txt`
- 同一条细节不要在两个文件里重复维护

## 压缩规则

- 先删重复，再补缺失
- 同一个含义只保留一处
- tool 顶层描述只写用途和边界
- 参数描述只写取值条件、保留条件、必填条件和必要映射
- 不是高频、稳定、可泛化的问题，不新增规则
- 对孤立 badcase 保持克制，接受少量误差，不要为了个例把 prompt 变厚

## 默认处理顺序

1. 退出误判、建议误执行、tool 误路由
2. 时间解析错、限定词漏保留、该执行未执行
3. 回复风格、播报自然度、回复稳定性
4. 少量边缘 badcase

## 输出要求

在输出一轮调优建议时：
- 先说明本轮目标错误类型
- 说明为什么该改 `system_prompt.txt` 或 `tools_call.json`
- 先总结删掉了什么重复，再说新增了什么
- 复测后说明收益和回退风险
